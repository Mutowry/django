In this article, I will be providing a walkthrough for the RAVEN: 1 vulnerable VM from Vulhub.com. This vulnerable VM is a Beginner/Intermediate boot2root machine. There are four flags to find and two intended ways of getting root. But I am showcasing one way to get to root I encourage you to find the other way!

Reconnaissance:

First thing let find the vulnerable machine on the network.

Command: netdiscover -r 192.168.99.0/24

In my case 192.168.99.12 was my target server.

Now lets do a quick initial nmap scan to see which ports are open and which services are running on those ports:

Command nmap -sC -sV -oA nmap_results.txt 192.168.99.12
  • -sC: run default nmap scripts
  • -sV: detect service version
  • -oA: output all formats and store in file nmap_results

So the system has SSH, HTTP, and RPC running on ports 22, 80, and 111 respectively. Interesting, but not too helpful just yet. Apache is version 2.4.10, which isn’t exactly new, but not so old as to make me think of an obvious exploit. Let’s see what information we can gather from the website itself first, then we can go and dig a bit deeper.

Enumeration:

We’ll start off by taking a look at the webpage running on port 80 with the title of Raven Security. As seen in the nmap previously conducted. And this is further confirmed by visiting the webpage itself in the browser.

Now let’s do some basic enumeration of hidden directories and files with gobuster against the raven site. This will be a very helpful starting point in finding clues for potential flags:

Command: gobuster dir -u https://192.168.99.12 -w /usr/share/wordlists/dirb/common.txt

Based on the results it looks like the site is running WordPress based on the /wordpress directory found. But it looks like we have a 301 status code meaning it was Moved Permanently. When we visit the page we are greeted with this:

So since WordPress is running let’s do a wpscan to enumerate for usernames. find any possible username that can be used later:

Command: wpscan --url https://192.168.99.12/wordpress/ -e u

From the scan, we are met with two usernames steven and michael. We will be using these later on in the walkthrough.

Initial Foothold:

From the WordPress scan, we got two usernames to work with steven and michael. Now let’s see if any of the enumerated usernames are possible usernames that can be used for SSH server running.

And after some testing, it can be found that basic brute-forcing/guessing work for the SSH login for the enumerate username michael. I found that the user michael SSH password is michael. This is a very weak obviously guessable password since I was able to guess it fairly easily. Without the need for any brute-forcing tool to brute force the ssh login such as with Hydra.

Flag Hunting:

Flag 1:

To find our first flag I decided to take a look into the var/www/html directory search the pages the exist there for any flags. Here is a list of HTML files I found:

  • index.html
  • about.html
  • service.html
  • team.html
  • contact.html

And I found that in the file named service.html it has the commented flag1 hidden within the source code:

Flag 2:

Now to move on to our second flag. We just simply need to move back a level from the var/www/html directory. To /var/www and simply type the ls command to see our second flag file(flag2.txt). And let’s plainly use the cat command to actually see the second flag:

flag3:

Now for the 3rd flag let’s look for MySQL credentials for the database. I found the key to my search in /var/www/html/wordpress/wp_config.php. The wp_config.php file is one of the core WordPress files. And it contains information about the database, including the name, host (typically localhost), username, and password:

Now that we have the username and password for the database let’s login:

mysql -u root -p’R@v3nSecurity’ -h 127.0.0.1

And after gaining access to the database now it is time to explore the database. And after some exploration I found that Flag 3 can be found in the wp_posts table in the database:

Commands:

  • show databases;
  • use wordpress;
  • show tables;
  • select * from wp_posts;

flag 4:

Now let’s go take a look at the wp_users table. And with further exploration of the table, it can be found that there is an unsalted password hash. For both michael and steven:

Let’s copy over steven’s hash since we already have michael’s password. To our Kali machine into a txt file. And let’s use john to crack copied over hashes.

john -wordlist=/usr/share/wordlists/rockyou.txt hash.txt

The whole process took a while but we did get the password for steven fairly quickly. Let’s see if we can SSH into his account with this password:

Success!
